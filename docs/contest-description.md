# Рекомендательная система Моей волны
Вам предстоит разработать модель, которая прогнозирует, прослушает ли пользователь трек в сервисе Яндекс Музыка.

В распоряжении участников находится открытый датасет Yambda, который содержит пользовательские взаимодействия с музыкальным контентом. Настоятельно рекомендуем ознакомиться с описанием датасета — там можно найти много полезных вещей.

Нужно решить задачу кандидатогенерации: для каждого пользователя необходимо предсказать список музыкальных треков, которые он с наибольшей вероятностью прослушает в будущем.

Для обучения моделей можно использовать весь общедоступный датасет, а итоговая оценка осуществляется на закрытых данных, собранных за более поздний временной период.

# Формат ввода
К задаче прикреплены файлы:
- data/users.csv — uid'ы пользователей, для которых нужно сделать прогноз. Они в точности совпадают с множеством uid'ов из Yambda-50M: range(100, 1_000_001, 100)
- data/baseline.ipynb — ipython-ноутбук с простым бейзлайном, записью ответа в требуемом формате и рекомендациями по решению
- data/test.csv — вывод бейзлайна
Формат вывода
Для проверки необходимо загрузить CSV-файл test.csv, который должен содержать ровно две колонки:

uid — целое число, ID пользователя. Множество uid должно совпадать с тем, которое указано в файле users.csv, повторений быть не должно
item_ids — прогнозируемые ID треков для соответствующего пользователя, разделённые пробелом.

Пример расчёта 
Recall@k
Recall@k для одного пользователя на Python:


def recall_at_k(true_ids: list, pred_ids: list, k: int) -> float:
    if len(true_ids) == 0:
        return 0.0
    
    top_k_preds = set(pred_ids[:k])
    true_set = set(true_ids)
    hits = len(true_set & top_k_preds)
    
    return hits / min(len(true_set), k)
Зачем нужно домножение на 1000
В метрике 
Recall@100
Recall@100 в этой задаче значимыми являются 5 цифр после запятой. Контест отображает лидерборд, округляя баллы до 2-х значащих цифр, поэтому для удобства мы домножаем на 1000:


score = round(recall_at_100 * 1000, 2)
Public / private scoring
По доброй традиции ML-соревнований мы разделяем тест на public и private. Это значит, что во время проведения соревнования участникам будут видны оценки только на public-множестве, а финальное распределение мест будет происходить по закрытому private-множеству.

Всё тестовое множество зафиксировано до начала соревнования.

Различия между public и private
Каждое из множеств public и private охватывает события за ~7 дней после датасета Yambda. Однако в private добавлен сдвиг на 1 сутки относительно конца Yambda, тогда как в public события взяты впритык к датасету. Наглядно на схеме:


Сценарий private больше приближен к индустриальным рекомендательным системам, т.к. в реальности есть задержка между доставкой логов, обучением модели на них и её деплоем в продакшн.
